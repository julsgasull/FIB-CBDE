Distributed databases

A distributed database (DDB) is a database where data management is distributed over several nodes in a network.

  • Each node is a database itself
  • Potential heterogeneity
  • Nodes communicate through the network

Distributed Architectures

  • Main objective: hide implementation (i.e., physical) details to the users
    • Data independency at the logical and physical level must be guaranteed
      • Inherited from centralized DBs (ANSI SPARC)
    • Network transparency
      • Data access must be independent regardless where data is stored
      • Each data object must have a unique name
    • Replication transparency
      • The user must not be aware of the existing replicas
    • Fragmentation transparency
      • The user must not be aware of partitioning

Material: Algorithms for Fragmenting Relations

En aquesta secció ens centrem en el disseny de bases de dades distribuïdes des de zero (és a dir, un disseny de top-down). Per tant, tenint en compte una base de dades i la seva càrrega de treball, analitzem com assignar dades a diferents llocs de manera que optimitzem un determinat conjunt de paràmetres (normalment, minimitzant el consum de recursos per al processament de consultes). Per fer-ho, s’han d’abordar dues qüestions principals: com fragmentar les dades i on assignar-les. Les dues qüestions estan fortament interrelacionades.

4.1. Data Fragmentation

La fragmentació de dades tracta el problema de trencar les relacions en trossos més petits i, per tant, de disminuir la unitat de treball del sistema distribuït. Hi ha moltes raons per fragmentar-se, però la idea principal és que una relació no és una bona unitat de processament.

Per exemple, considerem el cas d'un usuari que accedeix a una base de dades mitjançant una vista definida com a selecció sobre una relació R. Certament, aquest usuari mai no accedirà a cap instància de dades d'aquest subconjunt. En general, les aplicacions i els usuaris només accediran a un subconjunt de les relacions disponibles a la base de dades o, més concretament, a un determinat subconjunt d'algunes relacions.

Sense tenir en compte la fragmentació de les dades, tenim dues opcions: situar la relació en un sol node (i, per tant, augmentar els accessos remots i produir un coll d'ampolla potencial) o bé replicar-la a tots els nodes en què pot ser necessària la relació.

Tanmateix, es necessiten diferents subconjunts en diferents nodes i té total sentit, des del punt de vista del rendiment, col·locar aquells fragments susceptibles d'utilitzar-se conjuntament. Recordeu, això és el que es coneix com a localitat de dades. Com a resultat, la sobrecàrrega de la comunicació a través de la xarxa es minimitza i evitem la reproducció innecessària.

Considereu el nostre exemple en execució. Si volem aprofitar la localitat de les dades, ens interessarà fragmentar les dades i col·locar-les al node corresponent. Com s'ha comentat anteriorment, la nostra organització d'exemple es distribueix geogràficament i només ens interessa mantenir dades sobre projectes i empleats al node corresponent (ja sigui Hong Kong, Barcelona, Portland, Frankfurt o Glasgow, vegeu la figura 1).

Així, de tots els projectes, el node de Barcelona emmagatzemarà els projectes realitzats en aquesta ciutat (o millor dit, coordinats des d’aquesta ciutat), així com les assignacions i els empleats que treballen en aquests projectes. El mateix passarà amb cadascun dels altres nodes. En aquest cas, la relació del projecte s'hauria de fragmentar segons l'atribut de ciutat i segons el seu valor, situant la tupla en un node o un altre.

Quant a les relacions amb els empleats i les assignades, tingueu en compte que la fragmentació s'ha de realitzar d'una manera lleugerament diferent. Cada empleat i tupla assignada es col·locaran amb la seva tupla de projecte corresponent (per tant, primer hauríem de fragmentar la relació del projecte).

Tot i que beneficiar-se de la localitat de dades ha estat el motiu tradicional de la fragmentació, també ha aparegut un altre avantatge.

  • Descomposar una relació en fragments permet executar simultàniament diferents transaccions i també augmenta el rendiment del sistema.

És important destacar que quan es col·loquen fragments en diferents nodes, tots dos aspectes faciliten el paral·lelisme, que avui s’ha convertit en l’enfocament principal per tractar bases de dades molt grans.

Malgrat tot, la fragmentació també comporta certes dificultats.

  • D’una banda, fragmentar una relació pot comportar un rendiment inferior quan s’ha de recuperar i manipular diversos fragments en el moment de la consulta.
    • Aquesta situació pot passar quan hi ha requisits contradictoris que fan impossible la separació de fragments mútuament excloents.
  • D'altra banda, la comprovació de la integritat pot resultar costosa si dos atributs amb una dependència es divideixen en fragments (fins i tot en alguns casos senzills, depenent d'on s'assignin els fragments).
  • Per exemple,
    • suposem que en el nostre exemple actual, la seu de l'organització es troba a Hong Kong.
    • Des d’aquesta ubicació, haurien de recuperar periòdicament dades sobre tots els empleats. Segons la nostra discussió anterior, els empleats es distribueixen en tots els nodes segons el projecte en què treballin actualment.
    • Penseu en la següent consulta: identifiqueu els empleats que han estat treballant en almenys tres ciutats diferents.
    • Hem de recuperar tots els fragments dels empleats, assignats i del projecte en un node (suposem que les dades s’envien al node que emet la consulta; és a dir, Hong Kong), després reconstruir les relacions originals i unir-les correctament per respondre a aquesta consulta.
    • Per a aquest cas, la fragmentació és clarament un inconvenient; afecta el temps de resposta de la consulta, ja que incorrem en costos de comunicació entre nodes.
    • Sembla que la comprovació de la integritat també es pot complicar.
    • Penseu en un empleat que ha viscut a Barcelona i Glasgow. Actualment, treballa al projecte pB a Barcelona, però estava inscrit al projecte pG quan treballava a Glasgow.
    • Alguna cosa tan comú com comprovar la relació de clau estrangera entre l’assignat i l’empleat es fa més difícil, perquè les dades sobre l’empleat e es troben ara a Barcelona, mentre que l’assignació tuple e, pG es troba a Glasgow.
    • Una vegada més, hem d'incórrer en despeses de comunicació entre nodes per comprovar aquestes limitacions.

Fragmentar una taula és tan senzill com trobar alternatives per descompondre-la en taules més petites. És evident que tenim dos enfocaments principals:

  • la fragmentació horitzontal
  • la fragmentació vertical

Horitzontal

En el primer cas, s'utilitza un predicat de selecció per crear fragments diferents i, d'acord amb un valor d'atribut, col·locar cada tupla al fragment corresponent.

Per exemple, la fragmentació del projecte proposada al nostre exemple corrent s'adapta a aquesta categoria. Allà, els predicats que s’utilitzarien serien city = NAME_CITY (on NAME_CITY és cadascun dels valors dels atributs de la city: Barcelona, Hong Kong, Portland, Glasgow o Frankfurt).

Cada predicat donarà lloc a un fragment.

Vertical

Per contra, en cas de fragmentació vertical, es duen a terme diferents projeccions de la relació, cadascuna formant un fragment diferent.

Per exemple, considerem una vegada més la relació del projecte. Suposem ara que volem emmagatzemar els atributs de nom, ciutat i país a cada node, mentre que la resta de dades s’emmagatzemaran a la seu central de Hong Kong.

En aquest cas, es podria dur a terme una fragmentació vertical de la relació de la manera següent:

∏project(pno,name,city,country)∏project(pno,name,city,country)∏project(pno,name,city,country)﻿

and

∏project(pno,budget,category,productivityRatio,income).∏project(pno,budget,category,productivityRatio,income).∏project(pno,budget,category,productivityRatio,income).﻿

La necessitat de col·locar la clau primària a cada fragment vertical es justificarà adequadament més endavant. Tingueu en compte que diferents requisits poden conduir a diferents estratègies de fragmentació sobre la mateixa taula.

Fragmentació híbrida

Com és habitual, una tercera alternativa combina ambdues opcions i es coneix com a fragmentació híbrida. La fragmentació híbrida no és res més que l'estratègia de fragmentació horitzontal i vertical. En altres paraules, altres fragments fragmentaris produïts per una estratègia de fragmentació prèvia.

Per exemple, penseu en una combinació dels dos requisits discutits anteriorment: volem distribuir geogràficament dades sobre projectes, però només aquells atributs d'interès a cada node (els altres s'emmagatzemaran a Hong Kong tal com s'ha comentat anteriorment).

Per tant, podríem realitzar una fragmentació vertical, és a dir,

  • VF1 (∏project(pno,name,city,country)∏project(pno,name,city,country)∏project(pno,name,city,country)﻿) i
  • VF2 (∏project(pno,budget,category,productivityRatio,income).∏project(pno,budget,category,productivityRatio,income).∏project(pno,budget,category,productivityRatio,income).﻿)

VF2 es col·locarà al node de Hong Kong, mentre que VF1 es fragmentarà més aplicant la fragmentació horitzontal comentada anteriorment (és a dir, city = NOM_CITAT). Com a resultat, es produiran cinc fragments, cadascun d'ells es col·locarà al node corresponent.

En aquest document, les subseccions següents se centren en la fragmentació horitzontal i vertical i discuteixen dos aspectes principals:

  • el grau de fragmentació que podem assolir
    • El primer aspecte considera fins a quin punt és desitjable la fragmentació d'una relació (és a dir, quants subconjunts volem produir);
  • les regles de fragmentació per a la correcció.
    • el segon aspecte busca assegurar la correcció semàntica dels fragments.
    • Això últim s’aconsegueix mitjançant tres propietats:
      • Completeness (integritat)
        • Donada la relació R i un conjunt de fragments, qualsevol element de dades (ja sigui una tupla o un conjunt d’atributs) de R es pot trobar en almenys un fragment.
        • Per tant, les dades no es perden quan es fragmenten.
      • Disjointness
        • Donada una relació R i un conjunt de fragments, qualsevol element de dades col·locat en un fragment no es pot trobar en cap dels altres fragments.
        • Recordeu que la replicació de dades s'ha de considerar a posteriori, en l'etapa d'assignació, com una tasca independent de la fragmentació de les dades.
      • Reconstruction
        • Donada una relació R i un conjunt de fragments, la relació original sempre es pot reconstruir a partir dels fragments mitjançant operadors d’àlgebra relacional.

La fragmentació híbrida no es discuteix més, perquè correspon a una combinació imbricada de les dues estratègies i, per tant, és correcta si totes les estratègies de fragmentació posteriors aplicades són correctes.

4.1.1. Horizontal Fragmentation

La fragmentació horitzontal divideix una relació al llarg de les seves tuples. La forma de definir cada fragment és mitjançant predicats (és a dir, seleccions sobre qualsevol atribut de relació). La taula següent il·lustra una fragmentació horitzontal per a la relació del projecte en el nostre exemple. Aquí, cada fragment conté un subconjunt de les tuples de la relació.

Formalment, una relació R es fragmenta horitzontalment en n fragments mitjançant un predicat de fragmentació Ri=σFiR_i= σF_iRi​=σFi​﻿, 1≤i≤n1≤i≤n1≤i≤n﻿ on FiF_iFi​﻿ és el predicat de fragmentació que defineix el fragment RiR_iRi​﻿.

Normalment, representem cada fragment pel predicat utilitzat per crear-lo. En el nostre exemple:

  • HF1HF_1HF1​﻿: city = Barcelona,
  • HF2HF_2HF2​﻿: city = Hong Kong,
  • HF3HF_3HF3​﻿: city = Portland,
  • HF4HF_4HF4​﻿: city = Frankfurt,
  • HF5HF_5HF5​﻿: city = Glasgow.

El primer problema a tractar és quan un DBA hauria de fragmentar horitzontalment una relació determinada.

En general, un sistema distribuït es beneficia de la fragmentació horitzontal quan necessita reflectir dades distribuïdes geogràficament (cada node accedeix principalment a dades relacionades amb ell mateix),

  • per facilitar la recuperació i el paral·lelisme,
  • per reduir la profunditat dels índexs (ja que cada fragment té el seu propi índex, el nombre d’índexs augmenta però la seva mida es redueix)
  • per reduir la contenció.

En el nostre exemple, cada fragment és òbviament més petit que tota la relació (sempre que sabem que hi ha almenys una tupla a cada node).

Així, un índex sobre la clau primària (pno) donaria lloc a cinc índexs diferents (un per fragment), de mida menor (ja que cada índex només contindrà entrades per a les tuples d’aquest fragment).

La contenció es redueix clarament, ja que diversos usuaris mútuament excloents, que treballen en nodes diferents, poden accedir a diferents fragments simultàniament i no es produiran conflictes.

A més, les consultes sobre tota la relació es poden resoldre mitjançant paral·lelisme.

A continuació, hem de decidir fins a quin punt hem de fragmentar.

  • Tingueu en compte que la fragmentació pot anar d'un extrem (cap fragmentació en absolut) a l'altre (situant cada tupla en un node diferent).
  • A més, hem de saber quins predicats (sobre quins atributs) són d’interès a la nostra base de dades.
    • Per solucionar aquest problema, hauríem de comprovar quins predicats fan servir els usuaris (o aplicacions) que accedeixen a la base de dades.
    • Una regla general afirma que el 20% dels usuaris més actius produeix el 80% del total dels accessos a la base de dades.
    • Per tant, ens hauríem de centrar en aquests usuaris per determinar quins predicats hem de tenir en compte a la nostra anàlisi.
    • Per a cadascun d’aquests predicats, hem de realitzar els passos següents:
      1. Reunir totes les clàusules simples utilitzades en qualsevol predicat de consulta relacionat amb una relació determinada.

        A partir d’aquí suposem que un predicat és una expressió de la forma attrʘkattr ʘ kattrʘk﻿

        (on attr és un atribut de relació, és un dels següents operadors ≠,≤,≥,<,>,=≠, ≤, ≥, <,>, =​=,≤,≥,<,>,=﻿ i k és un valor en el domini d'attr).

      1. Per a cada conjunt de clàusules simples sobre el mateix atribut, completeu (d'acord amb la semàntica de l'operador de clàusules) afegint les clàusules complementàries que falten (és a dir, tots els valors de domini d'aquest atribut s'han de considerar com a mínim en un predicat).
      1. Per a cada conjunt de clàusules simples generades al pas anterior, determineu els conjunts de clàusules rellevants.

        Podem aplicar diferents criteris per trobar conjunts rellevants.

        Un enfocament típic consideraria un conjunt rellevant si cada clàusula d’aquest conjunt produeix un fragment al qual un usuari o una aplicació accedeix. Dit d’una altra manera, si té sentit utilitzar les clàusules d’aquest conjunt per fragmentar la relació.

      1. Per a totes les clàusules restants del pas anterior, tingueu en compte totes les combinacions entre clàusules sobre la mateixa relació i poseu-les en forma normal conjuntiva.
      1. A partir dels resultats del pas anterior, esborreu aquells predicats complexos que semànticament no tenen sentit.

        Aquí es poden utilitzar regles basades en el raonament en lògica de primer ordre per detectar predicats contradictoris.

      1. Entre els predicats restants, seleccioneu els predicats de fragmentació.

        Finalment, el dissenyador podria decidir afinar el resultat obtingut.

      Exemple

      Per exemple, considerem de nou la relació del projecte, que encara no s'ha fragmentat. I tingueu en compte les següents consultes realitzades pel 20% dels usuaris més actius:

      Q1: SELECT * FROM project WHERE city = ‘Barcelona’
Q2: SELECT * FROM project WHERE city = ‘Glasgow’ AND budget > 10000
Q3: SELECT * FROM project WHERE city = ‘Frankfurt’

      Ara, apliquem l'algorisme anterior pas a pas:

      → Pas 1

      Primer, reunim totes les clàusules emeses en qualsevol d'aquestes consultes. Així, obtenim: city = 'Barcelona’ , city = 'Glasgow', city = 'Frankfurt' i budget > 10000.

      → Pas 2

      Hi ha dos atributs implicats en aquestes clàusules: city i budget.

      Sabem que els valors del domini de la city són {Barcelona, Hong Kong, Frankfurt, Glasgow i Portland}, i el valor de budget oscil·la entre 200020002000﻿ i 450000450000450000﻿.

      Segons la semàntica de cada operador, complementem els predicats que falten de la següent manera:

      • Per a les igualtats, hem d’afegir un predicat per a cada valor del domini.

        Per tant, per a l’atribut city, hem d’afegir city = ‘Portland’ i city = ‘Hong Kong’.

      • Per als intervals, hem de completar l'interval.

        En el cas de l'atribut budget, hem d'afegir budget <= 10000.

      → Pas 3

      Ara hem de saber si cada predicat és rellevant per a almenys un node.

      La nostra organització està distribuïda geogràficament, de manera que tots els predicats sobre l'atribut de city són rellevants.

      Quant a l'atribut pressupostari, sabem que el budget > 10000 és rellevant a Glasgow, però suposem que budget <= 10000 no resulta interessant per a cap dels altres nodes.

      Per tant, la majoria de nodes no es beneficiaran d'aquesta fragmentació i, en conseqüència, es descarten tots dos predicats sobre el budget.

      → Pas 4

      Per a la resta de predicats, generem totes les combinacions.

      Tanmateix, només es mantenen predicats sobre l'atribut de city i no té sentit combinar-los, ja que s'exclouen mútuament (tingueu en compte que és impossible que una tupla tingui dos valors diferents per al mateix atribut).

Basics of Query Optimization

SQL es un lenguaje declarativo mediante el cual los usuarios expresan sus necesidades de información. Al ser un lenguaje declarativo, los usuarios declaran los datos que quieren obtener, pero no la forma en que debe recuperarse del sistema de gestión de bases de datos (DBMS).

Dada una declaración SQL, el objetivo del procesamiento de consultas es recuperar los datos necesarios para lograr esa declaración. Se llevan a cabo los siguientes pasos:

  1. Validación: esto implica la validación de sintaxis y esquema, pero también responsable para expandir vistas y verificar permisos.
  1. Optimización: se aplican ciertas reglas y heurísticas para generar con el plan de acceso más económico (a ejecutar en el siguiente paso).
  1. Ejecución: Es la ejecución del plan de acceso que se decidió en el paso anterior. Se refiere principalmente al acceso al disco y la transformación de datos aplicada (por medio de algoritmos, como ordenar o unir) que proporcionan los datos deseados.

Dada una consulta, hay muchas estrategias que un DBMS puede seguir para procesarla y producir su respuesta. Estas estrategias varían según los algoritmos disponibles y recursos utilizados (por ejemplo, disco y memoria). Todos ellos son equivalentes en términos de su producción final, pero varían en su costo, es decir, la cantidad de tiempo que necesitan correr. Esta diferencia de costo puede ser de varios órdenes de magnitud. Así, todos los DBMS tienen un módulo que examina diferentes alternativas y elige el plan que necesita la menor cantidad de tiempo. Este componente se llama consulta optimizador.

"Un optimizador de consultas traduce una consulta en una secuencia de operadores físicos que el motor de ejecución de consultas puede llevar a cabo directamente. El resultado del optimizador se denomina plan de acceso a consultas. El objetivo de la optimización de consultas es obtener un plan de ejecución eficiente en términos de medidas de rendimiento relevantes, como el uso de memoria y el tiempo de respuesta a las consultas". Evaggelia Pitoura

El optimizador, como se representa en la Figura 1, es un módulo dentro del DBMS;

  • entrada es una consulta SQL
  • su salida es un plan de acceso expresado en un lenguaje de procedimiento dado

Su objetivo es obtener el mejor algoritmo de ejecución posible.

Para hacerlo, el optimizador se basa en la siguiente información del catálogo:

  • Estadísticas de contenido
    • como número de tuplas, tamaño de atributos, etc.
  • Estructuras de almacenamiento disponibles
    • particiones y vistas materializadas.
  • Estructuras de acceso disponibles
    • árbol B e índices hash
  • Algoritmos aplicables: principalmente para unir y ordenar.

En general, un DBMS no encuentra el plan de acceso óptimo, pero obtiene una aproximación (en un tiempo razonable). Encontrar el óptimo es computacionalmente difícil (NP-complejo en el número de relaciones involucradas), lo que puede resultar en costos más altos que simplemente recuperar los datos. Por tanto, los DBMS utilizan heurísticas para llegar a una aproximación, es decir, en ocasiones no obtienen el óptimo, pero suelen estar cerca de él.

Es importante saber cómo funciona el optimizador para detectar desviaciones y corregirlas, cuando sea posible (por ejemplo, agregar o eliminar ciertos índices, particiones, etc.). Esta es la base de lo que se conoce como ajuste físico, la tarea principal de cualquier DBA, que consiste en desplegar el mejor escenario (por ejemplo, crear índices, vistas, agrupamiento, actualización de estadísticas, etc.) que produzca el mejor plan de acceso.

Centralized Architecture

Es crucial comprender completamente la optimización de consultas en entornos centralizados antes de abordar el mismo problema en sistemas distribuidos, ya que los mismos principios (con algunas extensiones) se aplican para la optimización de consultas distribuidas.

Por lo tanto, en esta sección, repasaremos los conceptos básicos de la optimización de consultas. Por ahora ignoraremos la presencia de una red, pero los datos pueden estar en diferentes discos en la misma máquina, siempre y cuando no usemos ningún tipo de paralelismo.

La optimización de consultas se presenta típicamente como si se ejecutara en tres pasos secuenciales (es decir, optimización semántica, sintáctica y física).

→ Semantic Optimization

Este es el primer paso que debemos dar y consiste en transformar (es decir, reescribir) la sentencia SQL en una equivalente de menor costo, considerando:

  • Restricciones de integridad: definidas en el esquema (por ejemplo, comprobaciones, unicidad, etc.)
  • Propiedades lógicas: algunas características lógicas básicas se pueden utilizar en este paso para explotar aún más las restricciones de integridad del esquema (por ejemplo, transitividad, subsunción, etc.)

En esta fase, el DBMS aplica transformaciones a una consulta determinada y produce una consulta SQL equivalente destinada a ser más eficiente. Por ejemplo, el predicado se puede estandarizar en la forma normal conjuntiva (CNF) (con servicios de razonamiento bien conocidos aplicados), las consultas anidadas se pueden aplanar en algunos casos, y similares. El objetivo principal es encontrar consultas tautológicas, consultas con forma incorrecta o consultas contradictorias (siempre desde el punto de vista de las restricciones de integridad). Tener una forma incorrecta significa que hay una mejor manera de escribirlos desde el punto de vista del desempeño, mientras que contradictorio significa que su resultado será el conjunto vacío. Es importante tener en cuenta que las transformaciones realizadas en esta fase solo dependen de las características declarativas (es decir, estáticas) de las consultas y no tienen en cuenta los costos de consulta reales para el DBMS específico y la base de datos en cuestión. En otras palabras, la entrada de este módulo es una consulta SQL y un esquema (con un conjunto de restricciones de integridad) y produce una consulta SQL optimizada equivalente sobre el mismo esquema.

La aplicación de la transitividad y la replicación de cláusulas sobre las equivalencias es un ejemplo típico de una transformación realizada en esta fase de la optimización para hacer frente a consultas incorrectas. Esto puede parecer un poco ingenuo, pero permite usar índices sobre ambos atributos. Por ejemplo, si el optimizador encuentra "a-b AND a=5", lo transformará en "a-b AND a=5 AND b=5", de modo que si hay un índice sobre "b", también se puede tomar en consideración en fases posteriores Otro ejemplo de optimización semántica para consultas incorrectas es la eliminación de disyunciones. Por ejemplo, podemos transformar la disyunción de dos igualdades sobre el mismo atributo en un "IN" (por ejemplo, "a — I OR a = 2" también se puede expresar como "un IN (1,2)"). Reducir el número de cláusulas en el predicado de selección de esta manera también podría reducir su costo de evaluación. Sin embargo, tal transformación no se detecta fácilmente en predicados complejos y solo se puede realizar en los casos más simples.Ahora, considere la siguiente relación:

CREATE TABLE people 
(
	id INT PRIMARY KEY, 
	age INT CHECK (age>18 AND age<65), 
	weight FLOAT
);

Cualquier consulta (parcial o completamente) que viole una restricción de integridad (ya sea del modelo o del esquema) debe identificarse como una consulta contradictoria. Por ejemplo:

Q1 = SELECT * FROM people WHERE id IS NULL
Q2 = SELECT * FROM people WHERE age = 66
Q3 = SELECT * FROM people WHERE weight>50 AND weight<45

Todas estas consultas recuperarían el conjunto vacío por diferentes razones. Por ejemplo, QI está solicitando valores NULL en un atributo definido como clave principal (es decir, violando una restricción de modelo que dice que las claves primarias no pueden ser NULL); Q2 viola una restricción de integridad de esquema (definida con un CHECK), mientras que Q3 viola las reglas lógicas con respecto a los números: un flotante no puede ser mayor que 50 y menor que 45 al mismo tiempo.

→ Syntactic Optimizationç

Consiste en traducir la consulta de SQL a una secuencia de operaciones algebraicas. Puede haber muchas secuencias de operaciones algebraicas correspondientes a la misma consulta SQL. Si es así, el optimizador elegiría el que tenga un costo mínimo, mediante heurística. Estas secuencias generalmente se representan en álgebra relacional como fórmulas o en forma de árbol (comúnmente conocido como árbol sintáctico). La interpretación del árbol es la siguiente:

  • Nodos, de los que distinguimos tres tipos:
    • Raíz: representa el resultado final de la consulta y, por lo tanto, es único.
    • Interno: Representa operaciones algebraicas (por ejemplo, uniones, proyecciones, uniones, etc.).
    • Hojas: Representan accesos a mesa (o relación).
  • Bordes: Representan flujos de datos. El nodo al que apunta la flecha requiere los datos producidos por el nodo ubicado en la cola de la flecha.

El objetivo de esta fase es reducir, tanto como sea posible, la cantidad de datos que fluyen desde las hojas hasta la raíz determinando el orden entre operadores. Deja los datos leídos de las tablas y los nodos internos transforman los datos leídos.

Por lo tanto, esto se puede lograr principalmente de dos maneras: (i) reduciendo el número de atributos lo antes posible y (ii) reduciendo el número de tuplas lo antes posible. Desde el punto de vista del álgebra relacional, (i) implica proyectar lo antes posible y (ii) significa seleccionar lo antes posible.

Proyectar y seleccionar lo antes posible es claramente deseable si queremos reducir la cantidad de datos que fluyen a través de las operaciones algebraicas tanto como sea posible. Sin embargo, tenga en cuenta que menos datos no siempre implica un plan de acceso más eficiente.

Los índices, clústeres y otras estructuras de acceso solo son accesibles a nivel de tabla (es decir, nivel de hoja) y nodos internos (es decir, operación interna atores) no se benefician de ellos. Sin embargo, las estructuras de acceso y los algoritmos de unión / clasificación no se consideran en esta etapa sino durante la optimización física.

Empujar selecciones y proyecciones en el árbol sintáctico es una heurística que normalmente conduce al óptimo, pero no siempre.

El optimizador comienza a crear todos los árboles sintácticos alternativos aplicando estas dos reglas:

  1. Propiedad conmutativa de las combinaciones: es decir, cambiar el orden de las entradas de una combinación determinada.
  1. Propiedad asociativa de las combinaciones: es decir, cuando se unen más de dos relaciones, conmutar las ramas de combinación de dos en dos.

Como ya sabe, las uniones son el operador relacional más caro y, por lo tanto, el orden de unión puede tener un gran impacto en el rendimiento. Por ejemplo, considere las dos combinaciones siguientes: A B X C y el siguiente escenario:

  • El 90% de las tuplas en B coinciden con al menos una tupla de A,
  • B y A se unen a través de una relación FK-PK (de A a B) y estadísticamente, sabemos que esta relación relaciona cada tupla en B con 10 tuplas en A, en promedio,
  • Solo una tupla de B coincide con cualquier tupla de C (y sabemos que coincide exactamente con una),
  • A tiene millones de tuplas, B tiene miles de ellas y C solo contiene algunos cientos.

CAP Theorem Revisited

  • The CAP theorem is not about choosing two out of the three forever and ever

  • Without partitions: CA
  • Otherwise...

Column-Oriented Databases

  • Column-oriented DBs are inherently aligned with decisional systems
    • A must for Data Warehousing! ~ Terabytes
    • A must for read-only Big Data systems ~ Petabytes
  • Two main types:
    • Relational Column-Oriented DBs (aka NewSQL)
      • First system: C-Store
      • Industrial examples: MonetDB, HP Vertica, SAP Hana, Oracle in-memory column store, MariaDB ColumnStore PostgreSQL Zedstore... in general, ANY relational database provides, in one way or another, a columnar engine

  • Column-Oriented Specific Features:
    • Data model
      • Pure Vertical Partitioning
      • Tuples are identified by their position (no PK needed to be replicated in each fragment)

      • Remove variable size records and work with fix-sized records (dictionaries or bitmaps needed)
      • Column-specific compression techniques
    • Specific query processing
      • Late materialization: apply as many processing operators to the vertical fragments before joining them

Data Model: Compression

  • Each column is not store as a regular file, but as a compressed file
  • In these DBs, the compression main objective is not reducing data space but reducing I/Os
    • Yet, data in columnar format is more compressible than data stored in rows
      • High data value locality (less value entropy)

    • Heavy weight compression algorithms (e.g., Lempel-Ziv)
      • In general, not that useful but it might be if there is a (huge) gap between memory bandwidth and CPU performance

      • Improves performance by reducing CPU cost
      • Decompression is needed in front of bitwise AND / OR

Data Model: Reconstructing Records

  • From each table, 3*N vectors are generated (N=#attributes)
  • To reconstruct the original tuple, the original vertical fragmentation used the PK
  • Alternatively, in column-oriented DBs each record has the same position in all the columns (vectors) created from that table
    • Thus, column-oriented DBs do not join fragments to reconstruct the record
  • Relevantly, a certain order might benefit a column and harm another
    • If necessary, in the presence of replicas, each replica might use a different order (however, the order must be the same for all fragments of a table in a given replica)

Data Model: Create Vertical Partitioning

  • Creating a fragment per column, in general, is suboptimal. Thus, most advanced DBs allow the user to define partitions (each, potentially, as a set of columns)
  • In these cases, finding the optimal vertical partitioning is a problem that must take into account the query workload
    • Identify correlations in the query workload and group those columns with high affinity (i.e., frequently queried together)
    • For m non-key attributes the search space is B(m), i.e., the mth Bell number, which counts the possible partitions of a set of m elements
    • For large numbers, B(m) ≈ m^m
  • Two main approaches to compute the optimal partitioning
    • Grouping
      • Attribute affinity matrix
      • Clustering algorithms
      • ...
    • Splitting
  • [RECAP] Partitioning must guarantee
    • Completeness
    • Disjointness (for the sake of performance, it might be sacrificed)
    • Reconstruction
